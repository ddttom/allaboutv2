# MX Spell Checker — Manual

**MX-aware spell checking for markdown files.**

---

## What It Does

`npm run mx:spell` checks the prose in a markdown file for spelling errors. It understands the MX ecosystem — terms like "maxine", "reginald", "gestalt", "yaml", and "frontmatter" are never flagged.

Both British and American English are valid. "Colour" and "color" are both correct. Only genuine typos — words wrong in both dictionaries — are flagged.

Three modes:

- **Report mode** (default) — Lists what's wrong. Doesn't touch the file.
- **Fix mode** (`--fix`) — Auto-replaces genuine typos with American suggestions. Leaves British and American spellings untouched.
- **Americanize mode** (`--americanize`) — Deliberately converts British spellings to American. Use when you want a single dialect.

---

## Quick Start

```bash
# Report mode — see what's wrong
npm run mx:spell -- path/to/file.md

# Fix mode — auto-replace typos
npm run mx:spell -- --fix path/to/file.md

# Americanize mode — convert British→American
npm run mx:spell -- --americanize path/to/file.md

# Help
npm run mx:spell -- --help
```

## What Gets Checked

| Content | Checked? |
|---------|----------|
| Markdown prose | Yes |
| YAML frontmatter | No (skipped) |
| Fenced code blocks | No (skipped) |
| Inline code | No (skipped) |
| Angle bracket content | No (skipped) |
| URLs | No (stripped) |
| HTML tags | No (stripped) |

The spell checker extracts prose only. Your YAML metadata, code examples, inline references, and angle bracket content are never touched.

## How It Works

### Dual-Dictionary Approach

**Both American English (en_US) and British English (en_GB) are valid.** A word correct in either dictionary passes. Only words wrong in both dictionaries are flagged as typos.

The logic:

1. Check every word against en_US and en_GB.
2. If valid in either — it's correct. No action needed.
3. If invalid in both — it's a genuine typo. Report with suggestions.

| Category | Rule | Example | Action |
|----------|------|---------|--------|
| Clean | Valid in en_US | "the", "color" | Nothing |
| Clean | Valid in en_GB | "colour", "specialised" | Nothing |
| Typo | Invalid in both | "colur", "arcitecture" | Report with suggestions |

### MX Vocabulary Filter

After aspell runs, any flagged words that appear in the MX word list (`.aspell-mx.pws`) are removed from the results. This word list is auto-generated by `npm run route:sync` from the concepts registry in [ROUTING.cog.md](ROUTING.cog.md).

Currently includes terms like: maxine, reginald, gestalt, joymaker, maxshine, yaml, frontmatter, npm, lifecycle, cli, aspell, and more.

## Fix Mode

```bash
npm run mx:spell -- --fix path/to/file.md
```

What happens:

1. The same report is printed
2. Genuine typos are replaced with American English suggestions
3. Replacements are **prose-only** — YAML, code blocks, and inline code are preserved
4. Case is preserved: `Arcitecture → Architecture`, `arcitecture → architecture`
5. British and American spellings are **left untouched**

### What Fix Mode Changes

| Before | After |
|--------|-------|
| arcitecture | architecture |
| colur | color |
| recieve | receive |

### What Fix Mode Does NOT Change

- Words inside YAML frontmatter
- Words inside fenced code blocks (` ``` `)
- Words inside inline code (`` ` `` )
- British spellings (valid — not errors)
- American spellings (valid — not errors)

## Americanize Mode

```bash
npm run mx:spell -- --americanize path/to/file.md
```

Opt-in mode for when you deliberately want American English throughout. This converts British spellings to their American equivalents.

### What Americanize Mode Changes

| Before | After |
|--------|-------|
| colour | color |
| Colour | Color |
| specialised | specialized |
| organise | organize |
| behaviour | behavior |

### What Americanize Mode Does NOT Change

- Words inside YAML frontmatter
- Words inside fenced code blocks (` ``` `)
- Words inside inline code (`` ` `` )
- Words already in American English

Combine with `--fix` to also auto-replace typos in the same pass.

## Adding Words to the Dictionary

If you have a domain term that keeps getting flagged, add it to the MX word list.

### Option 1: Add to an Existing Concept

Edit [ROUTING.cog.md](ROUTING.cog.md) — find the relevant concept and add the word to its `words:` array:

```yaml
- term: mx os
  words: [mx, npm, cli, yaml, md, frontmatter, lifecycle, todo, todos, ai, claude, aspell, YOUR-WORD-HERE]
```

### Option 2: Add a New Concept

If the word represents a new MX concept:

```yaml
- term: my-new-concept
  aliases: [alternative name]
  path: path/to/folder/
  definition: "One-sentence explanation."
  words: [myword, anotherword]
```

After either change, regenerate:

```bash
npm run route:sync
```

This updates `.aspell-mx.pws` automatically.

## Architecture

### Files

| File | Purpose |
|------|---------|
| `scripts/mx-spell.sh` | The spell checker script |
| `mx-canon/mx-maxine-lives/.aspell-mx.pws` | Auto-generated MX word list |
| `mx-canon/mx-maxine-lives/ROUTING.cog.md` | Source of concepts and words |

### How Prose Extraction Works

The script uses `awk` to parse the markdown:

1. Skip lines between first and second `---` (YAML frontmatter)
2. Skip lines between ` ``` ` markers (fenced code blocks)
3. Strip inline code (`` `...` ``)
4. Strip angle bracket content (`<...>` and unclosed `<...` to end of line)
5. Strip URLs, HTML tags, and markdown formatting characters
6. What remains is prose

### Dependencies

- **aspell** — `brew install aspell`
- **aspell en_US + en_GB dictionaries** — included with aspell on macOS
- **bash** — macOS default (bash 3 compatible, no associative arrays)
- **perl** — for fix/americanize mode replacements (always available on macOS)

## Relationship to the Prompt Enhancer

The MX Spell Checker and the Prompt Enhancer (route-decorator hook) share the same dual-dictionary approach and MX word list, but serve different purposes:

| Feature | Prompt Enhancer | Spell Checker |
|---------|----------------|---------------|
| Target | Live prompts | Markdown files |
| When | Every prompt, automatically | On demand (`npm run mx:spell`) |
| Output | Context injected into prompt | Terminal report + optional file fix |
| Prose awareness | No (checks raw prompt) | Yes (skips YAML, code blocks) |
| Dialect policy | Both valid, typos only | Both valid, typos only (+ opt-in `--americanize`) |
| Shared | MX word list, dual-dict | Same |

Both tools use `.aspell-mx.pws` — update once via `npm run route:sync`, both tools benefit.

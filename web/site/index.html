The Problem

40% of models on Hugging Face have fewer than 100 million parameters. You cannot detect which AI agent is visiting your site. User-Agent strings are spoofable. You must design for the worst agent — which means you're compatible with all of them.

🚫

Toast Notifications

Vanish before agents can process them. Humans miss them too.

🔍

Hidden Content

Pagination, infinite scroll, and SPAs obscure state changes from every parser.

🔒

Session Inheritance

In-browser agents inherit authenticated sessions. Banks cannot detect AI involvement.

🤖

Agent Diversity

CLI agents, browser automation, server-based, extensions, local models — all different capabilities.

The Solution: Cogs

A cog is a file — .cog.md, .cog.html, or .cog.json — that wraps any content in structured metadata. Humans get narrative. Machines get data. Same file serves both.

YAML Frontmatter

20+ metadata fields declaring what the content is, who it's for, and what machines can do with it.

  • Title, description, author, version
  • Status, category, audience
  • Tags, dependencies, certification
  • AI policy fields (editable, assistance level)

Schema.org Markup

Machine-readable structured data using established web standards — not proprietary formats.

  • Restaurant, Hotel, Store, Product
  • ProfessionalService, MedicalBusiness
  • JSON-LD embedded in the page
  • Validated against Schema.org specs

WebMCP Tools

AI agents can interact with your content through declared tool interfaces — query menus, check availability, switch languages.

  • Tool definitions in meta tags
  • Standardised input schemas
  • Executable from any agent
  • No custom integration needed

The Cogification Workflow

Transform any website into machine-readable, accessible, certifiable content in ten steps.

1

Analyse Source

Identify content type, languages, business domain, Schema.org type, existing metadata.

1.5

Enhanced Audit

Capture complete DOM tree, exact colours, computed typography, cached assets for offline work.

npm run cogify -- --target=https://example.com
2

Select Template

N-language business, standard doc, action doc, or info doc. Supports 1 to n languages with shared assets.

3

Generate Frontmatter

Comprehensive YAML metadata — title, description, version, status, tags, audience, certification, provenance.

4

Add Schema.org

Structured data that search engines and AI agents can parse directly. JSON-LD in the page head.

5

Implement WebMCP

Declare tool interfaces so AI agents can query menus, check reservations, toggle languages.

6

Ensure WCAG 2.1 AA

Skip navigation, semantic HTML, ARIA labels, colour contrast, alt text, keyboard navigation.

7

N-Language Architecture

Path-based routing (/en/, /es/, /de/), shared assets, hreflang tags, GDPR-compliant detection.

8

Add Certification

COG version, publication date, issuer, governance — making content certifiable and trusted.

9

Validate

W3C HTML, WCAG compliance (pa11y), Schema.org validation, YAML frontmatter parsing.

10

Register

Add to the cog registry, update the Reginald index, publish. Machine-readable, accessible, certifiable.

Standards First. Always.

MX doesn't replace established web standards. It builds on them. Everything that benefits SEO, accessibility, and usability also benefits MX. A well-built MX page is a well-built page.

Established Standards

Schema.org, semantic HTML, ARIA, Open Graph, Dublin Core, robots.txt, sitemap.xml — these come first.

MX Governance Layer

Where standards leave gaps — status, content type, AI policy, certification — MX metadata fills in.

No Duplication

MX never replaces what already works. It adds governance and lifecycle metadata that doesn't exist elsewhere.
